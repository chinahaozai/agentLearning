## Part 1 · 为什么要端云协同（不是非此即彼）★★

回扣 [轨道 Part 3.2](02-on-device-ai-track.md)：端和云各有不可替代的长处。

| | 端侧 | 云端 |
|---|---|---|
| 能力 | 弱（小模型）| **强**（大模型）|
| 隐私 | **强**（数据不出设备）| 弱（数据上传）|
| 延迟 | **快**（无网络往返）| 受网络影响 |
| 离线 | **可用** | 不可用 |
| 成本 | 用户算力，厂商省钱 | 厂商付推理费 |

**协同 = 各取所长**：简单/隐私/离线的留端上，复杂/算力重的甩云端。

但要诚实：**协同不是默认最优**。三种基线先想清楚该用哪个：

```
纯端：极致隐私/离线场景（如本地笔记助手）——能力够就别引入云的复杂度
纯云：重度能力依赖、不在意隐私/网络（如复杂创作）——别为了端而端
协同：既要隐私/离线兜底、又要在能用时上更强能力——大多数产品级 Agent
```

> 面试加分点：能说出"这个场景我**不**做协同，因为……"，比无脑"都要协同"更显成熟。

## Part 2 · 路由决策：按什么判断走端还是云 ★★★

### 2.1 四个判据维度

一个请求该走哪边，综合这几条判断：

| 维度 | 倾向端侧 | 倾向云端 |
|---|---|---|
| **任务复杂度** | 简单问答、格式化、本地数据相关 | 多步推理、长文生成、需要强能力 |
| **隐私敏感度** | 涉及私密数据（本地文件/聊天/健康）→ **强制端侧** | 公开/非敏感信息 |
| **网络状态** | 离线 / 弱网 → **只能端侧** | 在线且稳定 |
| **成本 / 电量** | 想省云端费用 | 端侧持续推理也耗电，长任务上云可能更划算 |
| **实时性** | 要求秒回、跟手 | 可以等几秒 |

> 注意"成本"不是单向的：端侧省的是云费，但**费的是用户的电**（回扣 [profiling Part 4]）。长任务在端上跑可能又慢又烫，这时云端反而是更好的体验。这种权衡讲出来很加分。

### 2.2 三种路由实现方式

| 方式 | 怎么做 | 优劣 |
|---|---|---|
| **规则路由**（推荐起步）| if-else 按上面的判据硬判 | 简单、**可解释**、可控；判据要自己设计好 |
| **置信度路由** | 端侧先试，输出"没把握"就升云 | 自适应；但**小模型置信度难测准**（见下）|
| **分类器路由** | 一个超小模型/分类器先判意图再分流 | 更智能；多一层、要训/调 |

> **置信度怎么测的坑**：别天真地"问小模型你有没有把握"——它自己也不靠谱。实务里更可靠的是：看输出的 logprob/perplexity 偏低、工具调用反复失败（[Agent/RAG Part 2.4]）、或检索没命中，这些**客观信号**触发升云，而不是信小模型的自我评估。

### 2.3 路由判断示例（规则路由）

```kotlin
enum class Route { LOCAL, CLOUD }

fun decideRoute(req: Request, net: NetworkState): RouteDecision {
    // 1) 硬约束优先：隐私 / 离线 一票决定
    if (req.touchesPrivateData) return RouteDecision(Route.LOCAL, "隐私数据，强制端侧")
    if (!net.isOnline)          return RouteDecision(Route.LOCAL, "离线，只能端侧")
    // 2) 能力判断：复杂任务上云
    if (req.estimatedComplexity > THRESHOLD) return RouteDecision(Route.CLOUD, "任务复杂，需大模型")
    // 3) 默认走端侧（快、省、隐私）
    return RouteDecision(Route.LOCAL, "默认端侧")
}
```

> 关键：**每个决策都带 `reason`**。这个 reason 要落进 trace（[收官项目 Part 4](04-edge-cloud-agent-capstone.md) 的 `route` 字段），既方便调试，也是面试时你能逐条解释"为什么这么判"的依据。**硬约束（隐私/离线）必须最先判、一票否决**。

## Part 3 · 降级与回退：协同系统的健壮性 ★★★

**失败不是例外，是常态**：没网、超时、端侧跑飞、云端限流。健壮的协同系统，核心就是降级设计。

### 3.1 两个方向的降级

```
云端不可用（断网/超时/限流）── 回退 ──▶ 端侧兜底
    （哪怕端侧能力差，也先给个可用结果，或排队等会儿重试）

端侧搞不定（置信度低/工具反复失败）── 升级 ──▶ 云端
    （前提：在线 + 非隐私，否则只能告知用户）
```

### 3.2 超时与取消

端侧可能慢（大模型小机器）、云端可能网络卡——**两边都要有 deadline**：

- 给每条路设超时；超时就走另一条或返回部分结果。
- 用户可取消（协程 cancel / 停止生成），别让一个慢请求卡死交互。

### 3.3 优雅降级（功能分级）

别"要么完美要么崩"。按能力分级，降级时退而求其次：

```
最佳：云端大模型 + 完整工具
  ↓ 云不可用
次好：端侧模型 + 本地工具 + 本地 RAG
  ↓ 端侧也吃力
兜底：模板回复 / 提示"当前网络受限，仅本地功能可用"
```

> **Android 类比**：这跟你做的**离线优先 + 网络降级**一模一样——有网拉最新、无网用本地缓存、再不行给占位/友好提示。你早就在 App 里写过这套，迁移过来就是端云降级。

## Part 4 · 一致性：端和云别变成两套东西 ★★

协同最容易烂的地方：端一套实现、云一套实现，**行为漂移、难维护**。三处要统一：

| 要统一的 | 问题（不统一时）| 对策 |
|---|---|---|
| **工具定义** | 端云各写一份工具 schema，行为不一致 | 工具 schema **一处定义**，端云共用同一份契约 |
| **行为契约** | 端云 system prompt 不同，人格/规则不一致 | 统一 system prompt / 行为规范 |
| **Trace 格式** | 端云 trace 拼不到一起，看不全一次请求 | 统一 trace/span 格式（[第3篇] + [收官 Part 4]），端云 span 能串成一条 |

另外两个要想到的：

- **上下文交接**：一个会话中途从端切到云（或反之），**之前的对话历史要带过去**，否则模型"失忆"。要设计上下文如何在两边传递。
- **能力/版本对齐**：端侧模型更新慢（要发版、用户不一定升级），云端更新快。要管理"端侧这版能做什么"的预期，别让路由把端侧搞不定的任务硬留在端上。

## Part 5 · 把路由也测了：协同系统的评测 ★

回扣 [Eval 篇](12-agent-eval-and-metrics.md)，协同系统多测这些：

- **本地命中率**：多少比例的请求留在了端上（越高越省云费、越隐私，但要不牺牲质量）。
- **端 / 云各自成功率** + **端到端延迟**（含路由 + 可能的降级重试）。
- **降级触发率**：多少请求发生了回退/升级——太高说明路由判据没调好。
- **综合成本 / 功耗**：协同方案 vs 纯云 vs 纯端，省了多少钱/电、质量差多少。

> 这正是 [PocketAgent M5](04-edge-cloud-agent-capstone.md) 的对比实验：**纯端 vs 纯云 vs 协同**三条线并排，用数据证明"协同在你这个场景到底值不值、最优配置是什么"。

## Part 6 · 常见坑 / 设计反模式

| 反模式 | 为什么坏 |
|---|---|
| 无脑全上云 | 丢了端侧的隐私/离线/省钱意义，那还做什么端侧 |
| 无脑全留端 | 小模型能力不够，复杂任务体验崩 |
| 路由判据散落各处、不可解释 | 面试官一问"凭什么这么判"答不上；线上出问题难查 |
| 没有降级 | 一断网/一超时整个 Agent 就崩 |
| 端云两套 prompt/工具，各自演化 | 行为漂移，越来越难维护 |
| 切换时丢上下文 | 用户感觉"它突然失忆了" |
| **隐私数据被路由到云** | **合规/信任雷区**——隐私必须是最先判的硬约束（Part 2.3）|

- **客户端入口型**（小红书那类）：你有客户端底子 + 端侧落地经验，正中下怀。
- **端侧 Agent 型**（面壁那类）：端侧部署 + 工具调用/编排 + 端云协同，全覆盖。
- **推理引擎型**（小米/阿里云那类）：补完 C++ + 量化 + profiling，够得着进阶。

> **一句话收尾**：这条轨道不让你去和算法博士拼数学，而是把你"懂手机"的老本行，叠上"会跑模型、会调工具、会协同云"的新能力——长成一个**别人难替代**的端侧 Agent 工程师。剩下的，是上手把 PocketAgent 做出来。
