# 端侧 AI 学习轨道（写给想转端侧大模型 / Agent 的 Android 开发者）

## 0. 这份文档怎么用

这是主路线图的一条**分叉轨道**，不是替代。

[Android 开发转 Agent 工程学习路线](01-agent-learning-roadmap.md) 那条主线，把你练成"能做 Agent 应用和 Agent 平台的全栈工程师"——方向对，但它的护城河偏低：你是在跟一大批后端、Python 转型的人抢同一个池子，Android 背景在那条线上只是加分项，不是门槛。

这条轨道补的是**另一半**：让模型**跑在端上（手机 / 芯片）**，做量化、加速、延迟 / 内存 / 功耗优化，以及端云协同的 Agent。这一半恰好吃你已有的 Android 功底，**别人卷不动你**。

取舍和主线一致：

- 核心概念讲透，不重要的不写。
- 能用 Android 生态类比就类比。
- 路线为主，难点才上代码（这份是"轨道图"，代码细节留给后续展开篇）。

适用前提（对应你的处境）：

- 已有 Android 工程经验，不排斥继续碰底层（性能 / NDK / C++）。
- **边做现职边慢慢转**，每周 5～8 小时，按里程碑推进、不按自然周。

带 ★ 的是重点：

- ★★★ **Part 3 心智模型** 和 **Part 7 把现职变成作品集** —— 一个是"看不看得懂端侧代码"的地基，一个是你转型最大的杠杆。
- ★★ **Part 2 真实 JD 画像**、**Part 5 核心能力** 和 **Part 6 分阶段路线**。

## 1. 为什么是你（护城河在哪）★

一句话：**端侧 AI = "推理" ∩ "移动端工程"，而后者你已经会了。**

做端侧的人，缺的从来不是"会调模型"，而是"能让模型在一台内存吃紧、要省电、还要不卡 UI 的手机上跑得动"。这后半句，就是你这些年在干的事。

| 端侧需要的能力 | 谁稀缺 | 你的现状 |
|---|---|---|
| 模型量化 / 推理原理 | 后端、客户端都缺 | 要补（Part 5.2/5.3）|
| 内存 / 带宽 / 功耗优化 | **极缺** | **你的主场** |
| NPU / GPU / 硬件加速 | 极缺 | 半个主场（你懂 GPU、Surface、硬件编解码）|
| JNI / NDK / C++ 集成 | 缺 | 看个人：NDK 工程好补，**C++ 多数人要补**（见[C++ 补齐计划](03-cpp-for-on-device-ai.md)）|
| 真机 profiling / 卡顿排查 | 缺 | **你天天做** |
| 在真实 App 里落地 | 缺 | **你天天做** |

> **关键直觉**：模型你**不用会训**。端侧岗要的是"会跑、会压、会加速、会塞进产品"，这是工程，不是科研。**从零训练 / 研究型微调**那条数学密集的赛道不碰；但"为上端而压缩 / 适配"的工程性微调躲不开，也学得会（见 Part 2.2）。

**Android 类比**：端侧推理工程 ≈ 你已经熟的**性能优化 + 硬件加速**，只是被优化的对象从"渲染管线 / 图片解码"换成了"一个神经网络的前向计算"。你排查过 jank、压过包体、调过 GPU 渲染、跟过功耗——同一套方法论，平移到"让一个 2B 模型在中端机上 30 token/s 还不烫手"。

## 2. 这条轨道对齐什么岗位：真实 JD 画像 ★★

> 本节基于真实招聘 JD（2026 年，BOSS 直聘端侧相关岗位）提炼，不是拍脑袋。**先看市场要什么，再决定学什么。**

### 2.1 "端侧 AI" 是一条光谱，不是一个岗位

同样挂"端侧"招牌，实际是四种差别很大的画像。**先认清你要瞄哪一型**：

| 类型 | 典型 JD / 公司 | 核心要求 | 对 Android 出身 |
|---|---|---|---|
| **客户端入口型** | 端智能技术专家 / 小红书（30-60K）| 精通客户端**其中一端**；C++ **加分**；端侧部署经验**优先** | ★★★ **最友好入口**：要的是客户端专家 + 端侧，AI 是加上去的 |
| **推理引擎型** | 高级 AI 开发 / 小米（50-70K）、端侧 AI sdk / 阿里云（40-60K）| 推理框架 / 编译器 / 算子、量化压缩、QNN / NeuroPilot、TFLite / ONNX、熟悉 Android/Linux | ★★ **护城河 / 进阶目标**：系统功底对口，缺"模型 + 引擎"那半 |
| **端侧 Agent 型** | 端侧 AI Agent / 面壁智能 | 端侧部署 + 量化 + **工具调用 / 任务编排**；精通 C++（内存 / 多线程 / 性能）| ★★★ **端侧 ∩ Agent 交集**，同时吃你两个兴趣 |
| **算法训练型** | 端侧 AI infra / 智元机器人（硕士）| RL、具身、Jax、**模型训练**、torchao | ✗ **避开**：数学密集的研究赛道，不是这条轨道的目标 |

> **最重要的认知**：别被"端侧"两个字唬住、以为都要会训模型。真正适合你的是**客户端入口型**和**端侧 Agent 型**——它们要"工程落地"，把你的 Android 功底当主资产；**推理引擎型**是补完功夫后的进阶目标；只有**算法训练型**才是该绕开的。

### 2.2 真实 JD 反复出现的关键词（你的学习清单照它对齐）

把多份 JD 的要求去重，高频出现的是：

- **语言**：C++（内存管理 / 多线程 / 性能分析）、Python。✓ Python 主线在学；**C++ 是最该补的硬门槛**（6 份 JD 里 4 份要，详见配套《[C++ 补齐计划](03-cpp-for-on-device-ai.md)》）。
- **推理框架**：TFLite / LiteRT、ONNX Runtime、TensorRT（对应 Part 5.3）。
- **模型压缩**：量化 / 蒸馏 / 剪枝（对应 Part 5.2）。
- **硬件**：高通 QNN / SNPE、MTK NeuroPilot、NPU 适配（对应 Part 5.4）。
- **系统**：熟悉 Android / Linux 系统编程。✓ **你的主场**。
- **端侧 Agent**：工具调用、任务编排（对应 Part 5.6，接主线 [第2篇](10-llm-api-and-tool-calling.md)）。

> 两个提醒：**①** JD 里的"微调"多指"为端侧适配 / 压缩模型"的工程性微调，不是从零训练——这部分躲不开，但学得会。**②** 这些岗薪资都是高级带（3-5 / 5-10 年），接得住的前提是拿得出"真机落地 + 设备指标"的硬货（正是 Part 6 的 M2–M4）。

### 2.3 一句话定位 + 最小胜任标准

一句话定位（可直接写进简历 / 面试开场）：

> 客户端工程背景，正在把能力延伸到**端侧大模型与 Agent**：模型量化与端上推理、端云协同的工具调用与 RAG，以及延迟 / 内存 / 功耗的工程优化。

**最小胜任标准**——最低要能独立完成：

> 在一台真机上，离线跑起一个量化后的小模型，做完 **CPU / GPU / NPU 三种后端**的延迟 / 内存 / 功耗对比，并把它接进一个真实 App 的功能里（哪怕很小）。

这比"我会调 OpenAI 接口"具体得多，也正好是上面那些端侧岗位面试官想听的。

## 3. 端侧推理的心智模型 ★★★

这一节几乎没代码，但它是后面所有内容的地基。**Android 开发初学端侧最容易在这四点上想歪**，想正了，后面读代码、看 benchmark 会非常顺。

## 3.1 端侧推理本质：一份权重 + 一个 runtime，跑在本机

云端调模型是"POST 一段 JSON 给远程接口"（见 [LLM API 与 Tool Calling](10-llm-api-and-tool-calling.md) Part 1）。端侧反过来：**模型权重文件下到设备本地，由一个本机的推理引擎（runtime）加载、做前向计算，全程不联网。**

```
云端：你的 App ──HTTP──▶ 远程 GPU 集群（模型在别人机器上）
端侧：你的 App ──▶ 本机 runtime ──▶ 本机 CPU/GPU/NPU（模型在用户手机上）
```

**Android 类比**：这跟你集成一个**本地 SDK / .so 库**没有本质区别——一个 native 推理引擎（llama.cpp / MNN / LiteRT 的 `.so`），加载一个资源文件（模型权重），给输入、拿输出。你早就干过"集成 native 库处理一段数据"这件事，只是这次数据是 token、库是推理引擎。

## 3.2 为什么要端侧（以及"端云协同"才是常态）

端侧不是为了炫技，是有硬收益：

| 收益 | 说明 |
|---|---|
| **隐私** | 数据不出设备（聊天、相册、健康、输入法）——很多场景这是硬要求 |
| **延迟** | 没有网络往返，首 token 更快，交互更跟手 |
| **离线可用** | 地铁、地下室、飞行模式照样能用 |
| **成本** | 算力用的是用户的手机，厂商省下大笔推理账单 |

但端侧打不过云端大模型的能力。所以真实产品多是**端云协同（hybrid）**：

```
                  ┌─ 简单 / 隐私 / 离线 ──▶ 端侧小模型（本机）
用户请求 ─▶ 路由 ─┤
                  └─ 复杂 / 算力重 ──────▶ 云端大模型（远程 API）
```

> **记住这条**：端侧岗位真正值钱的设计能力，是"**什么该留在端上、什么该甩给云**"的路由判断。这也是为什么主线学的那套云端 Agent（[第2篇](10-llm-api-and-tool-calling.md)）你**不能丢**——端云协同里"云"那半，就是它。

## 3.3 量化不是"压缩文件"，是用精度换体积和速度 ★

模型默认是 FP16（每个权重 2 字节）。手机塞不下、也跑不快，于是**量化（quantization）**：把权重用更少的比特表示（INT8 = 1 字节，INT4 = 0.5 字节），换更小的体积、更快的计算、更低的内存带宽，**代价是一点精度损失**。

一个模型的权重大致占多少内存，按"参数量 × 每权重字节数"估：

| 模型规模 | FP16 (×2) | INT8 (×1) | INT4 (×0.5) |
|---|---|---|---|
| 1B | ~2 GB | ~1 GB | ~0.5 GB |
| 2B | ~4 GB | ~2 GB | ~1 GB |
| 3B | ~6 GB | ~3 GB | ~1.5 GB |
| 7B | ~14 GB | ~7 GB | ~3.5 GB |

> 看懂这张表，你就明白为什么端侧基本只玩 **0.5B～3B + INT4**：一台 8GB 内存的手机，留给你的模型预算可能就 1～2GB。7B 在多数手机上是奢望。

**Android 类比**：量化跟你做过的**图片下采样 / 有损压缩**是同一种思路——`Bitmap.Config` 从 `ARGB_8888` 降到 `RGB_565` 省一半内存、画质略降；INT8/INT4 就是权重的"RGB_565"。区别是你要会判断"降到哪一档，质量损失还能接受"。

> 你只需要**会用、会评估**量化（选档、测质量掉多少），**不需要自己发明**量化算法。常见档：GGUF 的 `Q8_0`/`Q4_K_M`（llama.cpp 生态）、训练后量化方法 GPTQ / AWQ（知道名字、会用现成产物即可）。

## 3.4 端侧真正的瓶颈是内存带宽 / 功耗，不是算力 ★

云端聊"算力"，端侧聊**内存带宽和发热**。生成每个 token 都要把模型权重从内存搬一遍，**瓶颈常常是"搬数据"而不是"算数据"**（memory-bound）。这就是为什么 INT4 不只是省体积，还**直接提速**——要搬的字节少了一半。

而手机不能持续满载：温度一上来就**降频（thermal throttling）**，于是"刚开始很快、跑一会儿变慢"。

> 这几句话，恰好是你熟的领域：**带宽、缓存、发热、降频、电量**。纯算法出身的人对这些没感觉，你有。这就是面试里你能讲出深度、别人讲不出的地方。

## 4. 技术栈全景（要学什么）★★

端侧从上到下分几层。**从最上层（最贴近 Android、上手最快）往下学**，逐层深入：

| 层 | 选项 | 定位 / 何时用 |
|---|---|---|
| **① Android 官方高层 API** | ML Kit **GenAI** APIs（端侧 Gemini Nano，经 **AICore**）| 最省事。总结 / 改写 / 校对 / 图片描述等现成能力，仅限支持机型 |
| **② 自带模型的端侧引擎** | **MediaPipe LLM Inference**、**LiteRT / LiteRT-LM** | 想自己选小模型（Gemma 3 / 3n、Phi、Qwen）跑 chat / 自定义任务 |
| **③ 全控推理 runtime** | **llama.cpp**(GGUF)、**MNN-LLM**(阿里)、**ExecuTorch**(PyTorch)、**MLC-LLM**、ncnn(腾讯) | 跨平台、可控、可深度优化，端侧岗的硬功夫在这层 |
| **④ 硬件加速 / 厂商栈** | GPU(OpenCL/Vulkan)、高通 **QNN / AI Hub**、联发科 **NeuroPilot**；(NNAPI 已废弃) | 把推理压到 NPU，榨延迟 / 功耗 |
| **⑤ 端上应用工程** | 端侧 embedding + 向量库、tool calling、端云路由 | 把模型变成 Agent / RAG 功能（接回主线）|

**入门档 vs 进阶档**：

- **入门（先尝到正反馈）**：①②任选其一，在真机上把一个模型跑出第一个 token。
- **进阶（建护城河）**：③④，自己编译 runtime、切后端、测全套设备指标。

> **小模型清单**（认识就行）：Gemma 3 / **Gemma 3n**（Google 专为端侧 / 多模态做的）、Phi-4-mini、Qwen2.5 0.5B~3B、Llama 3.2 1B/3B、SmolLM2。优先选**有现成量化产物（.task / GGUF）**的，省得自己转。

## 5. 核心能力分解 ★★

仿主线"需要掌握的核心能力"，逐项拆开。前面带 ✓ 的是你**已有或主线已覆盖**的。

### 5.1 端侧 LLM 跑起来（最先做）

需要掌握：

- 选一个上手框架：**ML Kit GenAI**（最省事）或 **MediaPipe LLM Inference**（最灵活）。
- 模型下载 / 打包：`.task` 包或 GGUF 文件放进 assets / 下载到本地。
- 初始化推理引擎、设置 `maxTokens` / `topK` / `temperature`、拿到（流式）输出。
- 在真机上跑，而不是只在模拟器（端侧一切以真机为准）。

> 目标只有一个：**第一次在飞行模式下，让手机自己蹦出 token**。这一刻的正反馈会带你走完后面。

### 5.2 量化与模型格式

需要掌握：

- 格式：**GGUF**（llama.cpp 系）、LiteRT 的 `.task` / `.tflite`、ONNX、ExecuTorch 的 `.pte`。
- 量化档怎么选：`Q4_K_M` 通常是体积 / 质量的甜点；质量敏感再上 `Q5`/`Q8`。
- 怎么**评估**质量损失：拿一组固定输入，比量化前后的输出（接回主线 [Eval](12-agent-eval-and-metrics.md)）。
- 模型转换 / 量化工具链（用现成脚本即可，不必深挖原理）。

### 5.3 推理 runtime（硬功夫）

需要掌握：

- 选一个吃透：**llama.cpp**（生态最大、GGUF、好上手）或 **MNN-LLM**（国内大厂在用）或 **ExecuTorch**（PyTorch 官方、厂商后端全）。
- 用 **NDK** 在 Android 上交叉编译、集成进工程（JNI）。✓ NDK / JNI 你熟。
- KV cache、上下文长度、batch——理解它们怎么吃内存。
- 线程数、GPU 后端（Vulkan / OpenCL）开关对速度的影响。

### 5.4 硬件加速 / NPU

需要掌握：

- **delegate / backend 的概念**：同一个模型，可以让它跑在 CPU、GPU 还是 NPU 上。
- 高通 **QNN（AI Engine Direct）/ AI Hub**、联发科 **NeuroPilot**——把模型编译到 NPU。
- ✓ 你已有的 GPU / 硬件直觉直接复用。
- **现状提醒**：Android 旧的 **NNAPI 已废弃**，现在要么走 LiteRT 的 GPU/NPU 路径，要么直接对接厂商 SDK（QNN / NeuroPilot）。

### 5.5 端上 profiling（你的主场，别跳过）✓

需要掌握并**产出数据**：

- 指标：首 token 延迟、token/s（吞吐）、峰值内存、持续吞吐下的**发热 / 降频曲线**、每千 token 耗电。
- 工具：Android Studio Profiler、`systrace` / Perfetto、`adb` 电量统计、厂商温度接口。
- ✓ 这套你本来就会，只是测量对象换成推理。

> **这一节是你简历最硬的部分**。"把 X 模型在中端机上从 12 token/s 优化到 28 token/s，峰值内存降 40%，持续 5 分钟不触发降频"——这种话只有同时懂模型和懂手机的人说得出。

### 5.6 端侧 Agent / RAG 特有的坑（接回主线）

主线学过的云端 tool calling / RAG，搬到端侧会遇到新问题：

- **小模型的 tool calling 不稳**：参数容易乱填，常要用**结构化输出 / 语法约束（grammar / GBNF）**强制格式。✓ 概念见 [第2篇](10-llm-api-and-tool-calling.md) Part 4/5，端侧加一层"约束解码"。
- **端侧 RAG**：本地 embedding 模型 + 端上向量库（**sqlite-vec** / **ObjectBox** 向量检索），全程离线检索本地文档 / 笔记 / 聊天记录。
- 上下文窗口小、KV cache 吃内存——检索回来的内容要更精简。

### 5.7 端云协同架构（端侧岗的设计题）

需要能讲清：

- 路由策略：按任务难度 / 隐私等级 / 网络状态，决定走本地还是云。
- 降级：云不可用时回退本地；本地搞不定时升级到云。
- 一致性：端云两边 prompt / 工具定义怎么统一维护。

> ✓ "云"那半完全复用主线成果。这条轨道不是推翻主线，是**给主线长出端侧的另一条腿**。

## 6. 分阶段路线（边做边转，按里程碑非按周）★★

在职转型，**别按自然周排期**，按"能跑、能演示的增量"排里程碑，做完一个再下一个（仿 [收官项目](14-agentops-mini-platform-design.md) 的 M0–M5 写法）。每周 5～8 小时，整条轨道约 2～4 个月。

### M0 · 跑通第一个端侧模型（最快正反馈）

- **做什么**：用 ML Kit GenAI 或 MediaPipe LLM Inference，在**真机**上跑一个小模型，做个最朴素的本地 chat / 总结。
- **验收**：**飞行模式**下，手机离线蹦出第一个 token。
- **对应**：Part 5.1

### M1 · 换模型 + 吃透格式与量化

- **做什么**：自己下载 Gemma 3 / Qwen2.5 的量化产物换进去；同一模型测 INT8 vs INT4 两档的体积 / 速度 / 输出质量。
- **验收**：能用数据讲清"Q4 比 Q8 省多少内存、快多少、质量掉多少"。
- **对应**：Part 5.2

### M2 · 上真正的 runtime + 硬件加速（建护城河）

- **做什么**：用 **llama.cpp / MNN-LLM / ExecuTorch** 之一，NDK 编译进 Android 工程；切 CPU / GPU / NPU 后端，跑 Part 5.5 的全套 profiling。
- **验收**：一张**三后端对比表**（延迟 / 吞吐 / 内存 / 功耗）+ 一张 Perfetto / Profiler 截图。
- **对应**：Part 5.3 + 5.4 + 5.5

### M3 · 端侧 Agent / RAG

- **做什么**：给本地小模型加**一个能用的工具调用**（结构化输出约束）+ 一个**本地 RAG**（端上 embedding + sqlite-vec / ObjectBox 检索本地文档）。
- **验收**：离线状态下，模型能正确调一次本地工具、能基于检索到的本地内容作答。
- **对应**：Part 5.6（接 [第2篇](10-llm-api-and-tool-calling.md) / 主线 RAG）

### M4 · 端云协同 + 落地现职（临门一脚）

- **做什么**：写一个简单 router（本地 vs 云）；把一个**真实功能**推进你现职的 App（哪怕灰度 / 内部版）。
- **验收**：产品里真的上线了一个端侧或混合 AI 功能，且你有它的设备指标数据。
- **对应**：Part 5.7 + Part 7

> 走完 M0–M4，你不再是"在转型"，而是"已经在做端侧 AI"——这正是 Part 2 的最小胜任标准。

## 7. 作品集策略：把现职变成作品集 ★★★

这是你"边做现职边转"**最大的杠杆**，比任何业余 demo 都强。

业余项目再好，也是"我自己跑通了"；而**在真实产品里上线过端侧功能**，是"我在量产环境、真实机型分布、真实约束下交付过"——后者在面试里是降维打击。

**怎么找切口**（挑一个能塞进你现有产品的小功能）：

- 端侧**文本总结 / 智能回复**（离线、隐私友好，输入法 / 聊天 / 笔记类天然契合）
- 本地**语义搜索**（端上 embedding，搜聊天记录 / 笔记 / 商品 / 历史）
- **离线小助手 / FAQ Agent**（小模型 + 一两个本地工具）
- **AI 影像 / 相册**功能（如果你的 App 有图片场景）
- 把某个**现有的云端 AI 调用，改成端侧或端云协同**（省成本 + 提隐私，最容易说服团队立项）

**简历 / 述职怎么写**（对比一下分量）：

> ❌ 我做了个调大模型 API 的 demo。
>
> ✅ 在 XX App 落地端侧大模型功能：选型 Qwen2.5-1.5B INT4，基于 llama.cpp + NDK 集成，切 NPU 后端，首 token 延迟从 X 降到 Y，峰值内存控制在 Z，无网可用；并设计端云路由，复杂请求降级到云端。

> **顺带的好处**：这件事还会**反向加固你在现职的位置**——你成了团队里那个"懂端侧 AI"的人，转型期的收入和安全感都不受影响。这正是"边做边转"的最优解。

## 8. 面试表达方式

仿主线"面试表达方式"。不要只说：

> 我是 Android 开发，想转 AI。

可以说：

> 我有客户端工程背景，最近把能力延伸到了端侧大模型：在真机上跑通量化模型、做过 CPU/GPU/NPU 的延迟与功耗对比、并在产品里落地过一个端侧（或端云协同）功能。我的优势是同时懂模型推理和手机的内存 / 带宽 / 功耗约束——这块纯算法背景的人通常没感觉。

能展开的关键词（每个都对应本文一节，面试官追问你都答得出）：

| 关键词 | 你能讲的点 |
|---|---|
| 量化 | INT8/INT4、GGUF、体积 / 速度 / 质量的取舍（Part 3.3）|
| memory-bound | 端侧瓶颈是带宽不是算力、INT4 为何提速（Part 3.4）|
| 硬件加速 | delegate / NPU、QNN / NeuroPilot、NNAPI 已废弃（Part 5.4）|
| profiling | 首 token 延迟、吞吐、降频曲线、能耗（Part 5.5）|
| 端云协同 | 路由 / 降级 / 一致性（Part 5.7）|
| 端侧 Agent | 小模型 tool calling 的约束解码、端上 RAG（Part 5.6）|
